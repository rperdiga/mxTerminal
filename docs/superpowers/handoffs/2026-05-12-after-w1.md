## Architecture you need to know (gotchas)

These are the things that bit us during W1. A new session should know them up front.

### Two-extension layout

Each host gets its **own** Mendix extension folder:

```
<project>/extensions/
├── Concord10x/   ← Studio Pro 10.24.13 reads this
│   ├── manifest.json   { "mx_extensions": ["Concord.Host10x.dll"] }
│   ├── Concord.Host10x.dll
│   ├── Mendix.StudioPro.ExtensionsAPI.dll   (10.21.1)
│   ├── Concord.Core.dll
│   └── wwwroot/, skills/, skills-mac/, rules/, terminal.bundle.js
└── Concord11x/   ← Studio Pro 11.x reads this
    ├── manifest.json   { "mx_extensions": ["Concord.Host11x.dll"] }
    ├── Concord.Host11x.dll
    ├── Mendix.StudioPro.ExtensionsAPI.dll   (11.6.2)
    └── (same shared assets)
```

**Critical invariants:**
1. Each `manifest.json` lists exactly **one** DLL. Studio Pro 10.24.13's loader calls `.Single()` during hashing — multi-DLL manifests crash with `InvalidOperationException`. (Discovered during the spike on 2026-05-12.)
2. **Never co-deploy both folders to the same Mendix project.** Studio Pro will scan both, fail to load the wrong-version DLL, and either crash or refuse to recognize the extension. Use per-host deploy targets:
   - `MendixDeployTarget10x` — projects that get only `Concord10x/`
   - `MendixDeployTarget11x` — projects that get only `Concord11x/`
   - `MendixDeployTarget` — legacy fallback when the per-host override is unset
3. The wwwroot/skills/rules/skills-mac trees stay at the repo root (not moved into Core). `Concord.Core.csproj` references them via `Content Include="..\..\wwwroot\**\*"` with `<Link>wwwroot\%(RecursiveDir)%(Filename)%(Extension)</Link>` metadata so the folder structure is preserved in the output.

### ExtensionsAPI version drift

- **Studio Pro 10.24.13**: uses `Mendix.StudioPro.ExtensionsAPI 10.21.1` (verified — MCPExtension uses the same pin and works there).
- **Studio Pro 11.10+**: uses `Mendix.StudioPro.ExtensionsAPI 11.6.2` (matches v4.2.2).

The API surfaces differ in non-trivial ways. Most important: **menu API**:
- 10.x: `MenuExtension` is an **abstract base class** in `Mendix.StudioPro.ExtensionsAPI.UI.Menu`. You subclass it.
- 11.x: `IMenuExtension` is an **interface**. You implement it.

Same conceptual drift likely applies to `IDockablePaneExtension`, `IWebServerExtension`, `IApp`, etc. — each must be verified when porting Host11x classes to Host10x.

### MEF activation order

Every `[Export]` class in `Concord.Host11x` (and the new `ConcordMenuExtension` in Host10x) has a sentinel `[Import(typeof(Host11xEntry))] Host11xEntry? _entry;` (or `Host10xEntry`) field. This forces MEF to instantiate the Host{Nx}Entry **before** any other host class — so `HostContext.Initialize(...)` + `HostServices.Register(...)` run first. If you add a new MEF export to either host project, **don't forget the sentinel field** or you'll get uninitialized HostServices at runtime (silently in tests, observably in Studio Pro).

The CS0414 warning ("field is assigned but never used") is **intentional** — it's a sentinel, not a real reference. The `#pragma warning disable CS0649` in our code is technically the wrong warning code (should be `CS0414`); harmless but worth fixing in a polish pass.

### Build cache + Studio Pro cache

Studio Pro snapshots `extensions/<Concord*>/` into `<project>/.mendix-cache/extensions-cache/<guid>/` on first load. After that, it serves from the cache. If you deploy a new build while Studio Pro is running:
1. **File locks**: the running Studio Pro holds locks on the DLLs. xcopy reports "Sharing violation" and skips locked files. Build output goes into `extensions/` partially.
2. **Cache staleness**: even if the source `extensions/` updates, Studio Pro keeps serving the cached snapshot.

Workflow when iterating: close Studio Pro → wipe `.mendix-cache/extensions-cache/<guid>/` → rebuild → reopen. We have cache-refresh xcopy steps in each host's `DeployToMendix` target but they're best-effort.

### Naming smells (flagged during W1, not fixed)

- **Two `RunConfigurationInfo` types** in different namespaces:
  - `Terminal.Interop.RunConfigurationInfo` (Task 4, host-API record, fields non-nullable)
  - `Terminal.RunConfigurationSnapshot` (Task 8, snapshot DTO, fields nullable) — renamed from `Terminal.RunConfigurationInfo` to avoid the name collision but the conceptual overlap remains. Consider consolidating in W2.
- **Two host-injection mechanisms** running in parallel:
  - New: `HostServices.Register(...)` static registry (Task 4) — registered at startup but **nobody calls it yet**. Currently dead code.
  - Pre-existing: MEF `[ImportingConstructor]` injection + `Func<>` callbacks for `IRunStateProbe`, `IStudioProUiAutomation`. Still load-bearing.
  W2 should consolidate. Pick one — likely `HostServices` since it's the cleaner abstraction — and migrate the existing call sites.

## Things NOT to do

- **Don't** re-litigate the architecture decisions from the umbrella spec. They're settled. If user wants to change one, that's a new spec.
- **Don't** delete the umbrella spec or W1 plan even if they look outdated — they're history.
- **Don't** push the tag without explicit user confirmation. `git push origin v5.0.0-alpha.1` is a one-way commitment.
- **Don't** try to make Host10x feature-complete in a single quick session. That's W2's B2 strand and is a multi-hour port. If user asks for it, write a sub-plan first.
- **Don't** add a `<Compile Include>` glob anywhere — modern .NET SDK defaults work fine. Stick to `<Compile Remove>` for exclusions.
- **Don't** drop the per-host `_entry` sentinel fields when adding new MEF exports. They look unused but they're load-bearing for MEF activation order.
