## New gotchas (additions to the W1 handoff's gotchas list)

### Host10x defaulted `$(ExtensionsApi10xVersion)`

[`src/Concord.Host10x/Concord.Host10x.csproj`](../../../src/Concord.Host10x/Concord.Host10x.csproj) used to read `$(ExtensionsApi10xVersion)` from `Directory.Build.props` (gitignored). On a fresh clone with no props file — including every CI run and every new contributor — NuGet emits `NU1015: PackageReference item(s) do not have a version specified` and restore fails before anything else can run.

Fix shipped in `57c38d0`: csproj sets the property to `10.21.1` with `Condition="'$(ExtensionsApi10xVersion)' == ''"`, so a dev who pins a different Studio Pro 10.x NuGet version in their local props file still wins. Host11x already hard-codes `11.6.2` so it never had this issue.

**Implication for W2:** if you parameterize more host-specific NuGets later (e.g., a different Mendix DocsAPI version on 10.x), follow this same conditional-default pattern. Don't silently break the fresh-checkout flow.

### Windows MSBuild + `npm install --prefix` is fragile

The `BuildUi` target in [Host11x.csproj:31-38](../../../src/Concord.Host11x/Concord.Host11x.csproj#L31-L38) calls `npm install --prefix ../../ui --silent` when `ui/node_modules` doesn't exist. This works fine on macOS and on dev Windows machines (where `node_modules` typically already exists from prior builds). On a fresh `windows-latest` CI runner it fails with libuv error `-4058` (`ENOENT`) about 11 seconds in — almost certainly during esbuild's postinstall.

Workaround shipped in `a0d56ac`: CI installs UI deps via a dedicated `setup-node@v4` + `npm ci` step before `dotnet build`, so the MSBuild `BuildUi` target's `!Exists('ui/node_modules')` guard short-circuits the `npm install` call entirely. Only the `node esbuild.mjs` bundle step runs.

**Implication for W2:** if you add new MSBuild-driven shell commands that touch npm or node tooling, test them on a CI windows runner before assuming they work. The pattern "install deps in a workflow step, let MSBuild skip its own install" is now the precedent.

### GitHub Actions Node 20 deprecation (informational)

CI annotates a warning that `actions/checkout@v4`, `setup-dotnet@v4`, `setup-node@v4` run on Node 20, deprecated June 2026. Not urgent. Worth a bump to whatever version those actions publish supporting Node 24 before then.

## W2 scope — three strands

The umbrella spec's [W2 section](../specs/2026-05-12-concord-cross-version-mcp-merge-design.md#L127-L160) and the W1 handoff's [Section B](2026-05-12-after-w1.md#L118-L145) both cover this. Summary:

### B1 — Source-merge MCPExtension's tool surface into Concord.Core

Use `git subtree add --prefix=src/Concord.Core/Spmcp <MCPExtension-remote> main --squash` to import the standalone MCPExtension repo at `C:\Extensions\MCPExtension` into Concord. After the import, delete the bits that duplicate Concord plumbing (own MCP server, own pane/menu host, dev tooling), keep the ~84-tool implementations + handlers + utils + `backport-10x/`.

Refactor those tools to depend on **Core's `Interop` interfaces** instead of Studio Pro types directly. This is the biggest single chunk of W2 — touches ~14,500 lines across `MendixAdditionalTools.cs` and `MendixDomainModelTools.cs`.

Build `Concord.Core/Mcp/ToolCatalog.cs` that registers tools dynamically based on `TargetMode` (10.x vs 11.x) + family toggles. On 11.x register only the curated allowlist (table at [spec line 198-211](../specs/2026-05-12-concord-cross-version-mcp-merge-design.md#L198-L211)); on 10.x register the full surface.

### B2 — Port Host11x's UI-tier classes to Host10x against the 10.21.1 API

Mirror these Host11x classes into Host10x, adapted for the 10.x ExtensionsAPI surface:

- `TerminalMenuExtension` — note: 10.x uses `MenuExtension` abstract base class, 11.x uses `IMenuExtension` interface
- `TerminalPaneExtension` — same drift expected on `IDockablePaneExtension`
- `TerminalPaneViewModel`
- `TerminalWebServer` — same drift expected on `IWebServerExtension`
- `RunStateProbe` + `StudioProUiAutomation` — verify against 10.21.1 API

The MCPExtension repo's `backport-10x/` directory is the reference for what 10.x API calls look like. Read it for guidance during the port; don't copy code directly.

Once Host10x has its UI exports, the `ConcordMenuExtension.cs` placeholder from commit `63eee6d` gets replaced with the real menu.

### B3 — Consolidate the dual host-injection mechanisms

Today there are two parallel ways host services reach Core:

- **New**: `HostServices.Register(...)` static registry (W1 Task 4) — registered at startup but **nobody calls it yet**. Currently dead code.
- **Pre-existing**: MEF `[ImportingConstructor]` injection + `Func<>` callbacks (`IRunStateProbe`, `IStudioProUiAutomation`). Still load-bearing.

Once B1 + B2 land, migrate the call sites to `HostServices` and drop the MEF/`Func<>` chain. Task 15's "wire HostServices stubs to real services" then transitions from dead code to load-bearing.

## Branch strategy — decide before coding

Two options. **Default to (A) unless the user picks (B):**

**(A) Cut a new `feat/v5.0.0-w2-mcpx-merge` branch off `v5.0.0-alpha.1`.**
- W1 polish stays on `feat/v5.0.0-w1-foundation` and can either be merged to main now (it's small and CI-green) or kept until W2 lands.
- W2 gets its own clean branch with its own PR. Easier to review, easier to bisect.
- The alpha tag stays a clean anchor; W2 history doesn't muddy the W1 retrospective.

**(B) Keep going on `feat/v5.0.0-w1-foundation`.**
- Pro: one branch, fewer moving parts.
- Con: by the time W2 is done, the branch is 60+ commits and the "W1 foundation" label is misleading.
- Reasonable only if W2 is intended to ship as an extension of `5.0.0-alpha.2` rather than its own beta/RC line.

Whichever branch, **don't merge to `main` until W2's full surface is verified end-to-end on both Studio Pro versions.** Joe smokes manually; a passing test suite isn't the same as a working extension in Studio Pro.

## Things NOT to do (additions to W1's list)

- **Don't** disable CI to "fix it later." The two CI failures in this session were both legitimate signals about packaging issues (missing version default, Windows npm fragility) that would have bitten future contributors on a fresh clone. CI is now the contributor-onboarding canary.
- **Don't** add `Directory.Build.props` to git "to make CI work." It's gitignored for a reason: each dev's deploy targets differ. The fix is to default csproj properties conditionally, not to commit per-dev config.
- **Don't** use `cmd.exe`-only commands in MSBuild `Exec` tasks without an `OS` condition. The existing `DeployToMendix` targets do this correctly (one `Exec` per OS); follow that pattern if you add new build steps.
- **Don't** introduce `git subtree add` for SPMCP without first reading [W1 handoff's MCPExtension import note](2026-05-12-after-w1.md#L122-L124) — get the exact remote URL + branch from the MCPExtension repo, and confirm the squash strategy is what the user expects.
