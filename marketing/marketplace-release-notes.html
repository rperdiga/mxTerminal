Concord 4.2.2

The headline of this release is TOML hygiene + crash-safety cadence. Two real-world bugs and two rule sharpenings — every change driven by the first production Codex build using v4.2.1's three-CLI rules paths. v4.2.2 closes the gaps that build surfaced.

If you upgraded Concord from before v1.3.0 with Codex enabled, this release fixes the Error loading config.toml: invalid transport hard-block that prevented Codex from starting. If Codex's "External agent config detected" prompt has been pestering you on every project, that's gone too for Concord-managed projects. Plus two surgical rule edits: maia__reset is now correctly scoped to actual failures (not benign diagnostic responses), and save_all has an explicit 15-minute fallback for visual-polish phases where natural batch boundaries don't exist.

What you'll experience

If you upgrade an existing 4.2.1 install: open the Concord pane once after upgrading. The orphan-tools cleanup runs as part of Concord's normal Save cycle on the upgrade-apply path — no manual config-file surgery needed even if you previously had to edit ~/.codex/config.toml by hand. Same goes for the migration-prompt suppression: Concord stamps the per-project entry on first Save after upgrading.

If you're a CLI agent using v4.2.1's rules: the two rule clarifications (§2 maia__reset trigger sharpened, §12 time-based save_all) land automatically when Concord refreshes the always-loaded rules on first Save after upgrade.

If you're installing fresh on v4.2.2: same flow as any prior release. Drop the .mxmodule into your Mendix project, restart Studio Pro, open the Concord pane. All the wiring lands automatically.

What's new in 4.2.2

AreaChange
Hard-block fix Codex 0.128+ wouldn't start after v1.3.0 migration Pre-v1.3.0 Concord wrote per-tool sub-sections under [mcp_servers.mendix-studio-pro-actions] to ~/.codex/config.toml. v1.3.0+ renamed the section to concord-mcp but left the orphan [mcp_servers.<old>.tools.<X>] sub-sections behind. Older Codex tolerated this; Codex 0.128+ refuses to start with "invalid transport in mcp_servers.mendix-studio-pro-actions". v4.2.2's RemoveNamed now strips parent + all child sub-sections in one pass, including orphans where the parent was removed by an earlier failed migration. Affected users: anyone who upgraded across the v1.3.0 rename with Codex enabled. Concord Save on first open after upgrade resolves it automatically.
New Codex migration-prompt suppression for Concord-managed projects Codex 0.128+ shows an "External agent config detected" prompt on first entry to any project — offering to migrate Claude Code MCP servers / subagents / plugins / sessions. For Concord users that prompt is noise: Codex is already configured by Concord. v4.2.2 stamps the project under [notice.external_config_migration_prompts.project_last_prompted_at] with a far-future unix epoch (year 2099) when Concord wires Codex. Surgical: only the project being applied to is stamped; home-level prompt and other projects are untouched. Idempotent: re-apply is a no-op when the stamp is already future-dated.
Rules §2 Maia recovery ladder sharpened — maia__reset for actual failure only v4.2.1's introspection toolkit (maia__health, maia__busy, maia__ping) returns diagnostic information that the agent was pattern-matching as failure. Codex's first production run called maia__reset 51 times across two sessions despite zero actual bridge disconnects. v4.2.2 rule explicitly lists which response shapes are SUCCESS (not failure): maia__health with available: true for any transport, maia__busy reporting {busy: true}, maia__ping with non-zero latency_ms, maia__status with streaming: true or lost: true. maia__reset is for recovering FROM observed failure, not for prophylactic bridge hygiene.
Rules §12 Verification — time-based save_all fallback v4.2.1's "save_all + refresh_project after each batch" works during build phases but fails during pure visual-polish (re-running pg_write_page against existing pages, tweaking theme variables) — no batch boundaries means save_all is never triggered. CocktailDemo34 (2026-05-10) drifted 54 minutes without a save before a machine crash; the build survived only because Codex happened to save 3 minutes before the crash. v4.2.2 adds a hard rule: call save_all + refresh_project at least every 15 minutes of continuous work, even when no logical batch has completed. Closes the crash-safety gap on long polish loops.

Empirical baseline

Every change in v4.2.2 traces to a specific observation from the 2026-05-10 CocktailDemo34 Codex production run — the first end-to-end build using v4.2.1's three-CLI rules paths. The Codex run validated v4.2.1's headline claim (Codex consumes .codex/rules/concord-*.md via AGENTS.md @-imports exactly as Claude Code consumes its parallel set) AND surfaced the four follow-up gaps shipped in v4.2.2. Empirical, not theoretical.

Cross-platform note

Concord ships on Windows and macOS. Most features are platform-identical; the v4.2.2 TOML cleanup applies on both platforms (Codex 0.128+ runs on both). Maia integration is Windows-only — driving Maia from outside Studio Pro requires inspecting its WebView, which works on Windows but not on macOS (WKWebView requires host opt-in by Mendix). Concord ships a Mac-specific variant of the page-gen skill for that case.

Compatibility

  • Studio Pro: 11.10.0 or newer
  • Operating system: Windows 10 build 17763 (1809)+ · macOS 10.15 (Catalina)+
  • .NET runtime: 8.0 (bundled with Studio Pro 11)

Privacy

Concord is loopback-only. The Concord MCP server binds to 127.0.0.1; Maia integration uses Studio Pro's local WebView2 debug port. Nothing leaves your machine through Concord itself.
