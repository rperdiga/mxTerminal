Concord 4.2.0 — Release Notes (paste this)

Concord 4.2.0

The headline of this release is Maia bridge hardening. The Maia integration introduced in earlier releases is now reliable for long Claude Code sessions inside Studio Pro. A persistent CDP connection eliminates the per-call PowerShell + WebSocket churn that surfaced under load, a 10-second heartbeat defeats Chromium's WebView2 background-tab throttling, an auto-re-injection path recovers cleanly when the Maia panel reloads, and the response parser now surfaces the actual shape it received instead of swallowing the error. Plus a new optional Diagnostic logging toggle in Settings → Concord MCP for ground-truth debugging, and a focused rules-content iteration that closes six gaps surfaced by real-world build forensics.

If you ran into IOException: forcibly closed, Unknown handle, or poll() returned unexpected shape errors while driving Maia from Claude Code in a long session, this release is for you.

What you'll experience

If you upgrade an existing 4.1.x install: open the Concord pane once after upgrading. Concord's MCP server starts as before, the bundled Mendix skill packs and rules refresh, and the Maia bridge is now substantially more reliable on its own — there's nothing new for you to configure. The always-loaded build rules at <project>/.claude/rules/ are now split into three files (concord-build-rules.md for core operational discipline, concord-pages-and-themes.md for UI construction, concord-model-discipline.md for model rules) — the split is purely structural so the per-file size stays under Claude Code's performance threshold, content is byte-identical to what was in the single 4.1.4 file plus six new rules from the v4.1.5 iteration. Your project's CLAUDE.md managed block is regenerated to @-import all three.

If you want to see CDP traffic for a Maia issue: Settings → Concord MCP → Diagnostic logging. Off by default; flip it on, reproduce the issue, then read resources/terminal.log for [cdp] >> id=N / [cdp] << id=N traces. Off again when you're done.

If you're installing fresh: same flow, just without the "previous version" comparison. Drop Concord in, restart Studio Pro, open the pane — the wiring is automatic.

What's new in 4.2.0

AreaChange
Reliability Persistent CDP connection The Maia bridge now reuses a single CDP WebSocket and a single port-discovery result across every maia__* tool call for the lifetime of your Studio Pro session. Previously each call opened a fresh WebSocket and spawned a fresh PowerShell process to look up Studio Pro's debug port; under sustained load (a long-running Claude Code build firing dozens of maia__wait calls) this saturated the local CDP server and produced IOException: forcibly closed by remote host. The persistent connection eliminates that failure mode.
Reliability Auto-reconnect on WebSocket drop If the persistent connection does drop mid-call (e.g. Studio Pro restart), the bridge transparently reconnects once and retries the call before raising an error. Discovery cache invalidates on connect failure, so a Studio Pro restart self-heals after one failed call.
Reliability Auto-re-injection on WebView reload When the Maia panel reloads (panel close/reopen, "New chat" click, V8 GC), the JS-side bridge agent is gone. The transport now detects this case via a structured {__reinject:true} signal, reinstalls the agent, and retries the call once. The previous behavior surfaced confusing Unknown handle: <name> errors mid-session; those are now self-recovering.
Reliability Defensive poll() parser When Maia returns an unexpected response shape (rare but observed during layout-edit operations), the bridge now logs the actual shape it received instead of throwing the cryptic poll() returned unexpected shape: empty-string error. Genuine non-object responses surface a structured message naming the shape.
New Lost-handle discriminator If a WebView reload nukes the JS-side ticket but Concord still has the handle bound, maia__status and maia__wait now return a lost: true discriminator instead of looping. Callers can re-ask cleanly. Genuinely unknown handles (typo, never sent) still surface as a structured Unknown handle error.
Reliability CDP keep-alive heartbeat Every 10 seconds, the bridge fires a no-op CDP 1+1 evaluate over the persistent socket. This defeats Chromium's WebView2 background-tab throttling that otherwise drifts Maia's internal scan timer from 500ms to seconds when Maia is hidden behind another pane in Studio Pro. Heartbeat is gate-aware: if a long maia__wait is mid-flight, the heartbeat skips that beat instead of failing.
New Diagnostic logging toggle Settings → Concord MCP → Diagnostic logging. Default off. When on, every CDP request and response is traced to resources/terminal.log in the form [cdp] >> id=N bytes=… and [cdp] << id=N bytes=…. Use to ground a bridge-failure diagnosis; turn off again to keep the log lean. Windows-only (same gate as Maia integration itself).
New Build-rules iteration (v4.1.5) Six surgical edits to the always-loaded build rules, all from real-world build forensics: broader Maia recovery trigger covering all bridge error variants (not just "panel not visible"), hard cap on consecutive bridge failures (3 → STOP + escalate), layout-edit manual fallback as the default, mark-as-UI-resources soft-stop now fires before the first SCSS write (was deferred), save_all + refresh_project cadence rule documented, and a read-loop anti-pattern guard.
New Build-rules split into 3 files The rules content (15 sections, content unchanged from 4.1.4 + the six v4.1.5 additions) is now distributed across three files at <project>/.claude/rules/: concord-build-rules.md (core operational discipline), concord-pages-and-themes.md (UI construction), concord-model-discipline.md (model rules). Each file stays under Claude Code's per-file performance threshold. Section numbering remains globally unique (§1–§15) so cross-references resolve regardless of which file they're cited from. The CLAUDE.md managed block now @-imports all three.

Cross-platform note

Concord ships on Windows and macOS. Most features are platform-identical; two differences worth calling out:

  • Maia integration (and the Diagnostic logging toggle that supports it) is Windows-only — driving Maia from outside Studio Pro requires inspecting its WebView, which works on Windows (WebView2 + CDP) but isn't available on macOS (WKWebView requires host opt-in by Mendix). Concord ships a Mac-specific variant of the page-gen skill that hands the work to Maia through you instead of programmatically.
  • Hotkey-driven Studio Pro tools on Mac need Accessibility permission the first time — macOS prompts for it; you grant Studio Pro one-time access in System Settings → Privacy & Security → Accessibility.

Compatibility

  • Studio Pro: 11.10.0 or newer
  • Operating system: Windows 10 build 17763 (1809)+ · macOS 10.15 (Catalina)+
  • .NET runtime: 8.0 (bundled with Studio Pro 11)

Privacy

Concord is loopback-only — the Concord MCP server binds to 127.0.0.1; Maia integration uses Studio Pro's local WebView2 debug port. Nothing leaves your machine through Concord itself. The bundled skill packs and rules are static Markdown files copied into your project. The CLI agents you run inside Concord follow their own privacy contracts. The new Diagnostic logging toggle only writes to resources/terminal.log on your local disk — nothing is transmitted.
