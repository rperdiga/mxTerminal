Concord 4.2.1 — Release Notes (paste this)

Concord 4.2.1

The headline of this release is bridge introspection + rules iteration from production validation. v4.2.0 made the Maia bridge reliable; v4.2.1 makes the agent's discipline equally reliable and gives the CLI a small toolkit for asking the bridge "are you alive? is Maia generating? what's your state?" without burning a real call. Plus rules edits driven by the 2026-05-10 production-validation run that exposed gaps in the v4.1.5 ruleset.

Four new MCP tools — maia__busy, maia__ping, maia__health, maia__new_chat. A task-scoped Maia-bridge failure cap (was raw count, fired prematurely on three unrelated transient errors). An errors-before-run_app hard gate. A Maia-as-page-fixer tiebreaker before user homework. The seed-data self-service-button pattern codified after landing in production. Codex + Copilot rules paths lit up alongside Claude. Codex defaults-on with a first-run banner. All three v4.2.0-deferred carry-overs (singleton dispose on toggle, WebSocket test-seam, JS-side scan throttle) ship in this release.

What you'll experience

If you upgrade an existing 4.2.0 install: open the Concord pane once after upgrading. Concord's MCP server starts as before; the bundled skill packs and rules refresh automatically. You'll see a one-line notice if Codex was previously unticked and v4.2.1's default-on behavior re-enables it: "Codex MCP wires ~/.codex/config.toml (user-global, outside the project tree). Untick Codex in Settings if you'd rather not." Untick if you prefer; otherwise it's done. Your project's CLAUDE.md managed block is regenerated; if you ticked Codex or Copilot, new managed blocks land in AGENTS.md and .github/copilot-instructions.md respectively.

If you're a CLI agent author or you write rules: the always-loaded build rules at <project>/.claude/rules/ have four targeted edits this release — see the table below. The rules are now also installed in parallel .codex/rules/ and .github/rules/ folders when those CLIs are ticked.

If you're using Maia from a long Claude Code session: the four new maia__* introspection tools are available immediately. maia__health is the recommended first call when something feels off — it reports bridge state without sending traffic to Maia.

What's new in 4.2.1

AreaChange
New maia__busy Read-only DOM probe — "is Maia generating?". Returns {busy, reason: 'spinner-visible' | 'recent-dom-mutation' | 'idle', idle_for_ms, spinner}. No traffic to Maia. Use before maia__new_chat to avoid interrupting mid-stream.
New maia__ping Cheap liveness probe. Sends "ping" with auto-sentinel, waits up to timeout_sec (default 5s), returns {alive, latency_ms, response, timed_out}. Reuses existing send/wait infrastructure including the v4.2.0 lost-handle discriminator. Use before expensive maia__ask calls when bridge health is uncertain — fail fast instead of burning a 60-second timeout.
New maia__health Bridge-state introspection without traffic to Maia. Returns per-transport availability + last latency + reason, last probe time, in-flight handle bindings, forced tier, plus an embedded busy() snapshot. One-call diagnostic before any recovery decision — answers "is it me, the bridge, or Maia?" without round-tripping anything.
New maia__new_chat Programmatic click of Maia's "New chat" button — wipes Maia's panel context. Used in the recovery ladder between maia__reset and user-handoff. Always preceded by maia__busy per the rule (interrupting mid-generation can corrupt Maia's state). Optional task-boundary use too: when the next prompt would not meaningfully reference the prior conversation, a fresh chat keeps Maia sharp.
Rules §3 task-scoped failure cap Replaces v4.1.5's raw-count cap ("3 consecutive maia__* failures = STOP") with a task-scoped variant: count failures on the SAME logical operation. Three transient errors across three unrelated tasks is normal noise — v4.2.0's auto-reconnect handles each cleanly. Three failures on the same handle, prompt retry, or reset cycle without forward progress is genuinely stuck and should escalate. Production validation: the raw-count cap fired prematurely on the 2026-05-10 demo build and stopped an otherwise healthy run.
Rules §12 errors-before-run_app hard gate run_app is now gated by zero errors. Run ped_check_errors on every touched doc; if any has errors, do NOT call run_app. Fix at the model layer where errors are actionable; the runtime won't surface them usefully (they appear as runtime crashes, console spew, or pages that 500). Production validation: an agent looped run_app 19× against 2 unresolved errors before this rule existed.
Rules §3/§5 Maia-as-page-fixer tiebreaker After a single ped_update_document page-fix doesn't resolve the errors, the rules now route the page through Maia for one more attempt before user homework. Mirrors the §2 "Maia owns pages" doctrine. Non-page docs (microflows, entities, view entities) stay on the §5 single-shot rule — Maia can't reliably edit those.
Rules §8 seed-data self-service button Codifies the production-validated pattern: ProjectManage singleton entity + NeedProjectSeedData Boolean + a visibility-bound button on the home page + a Playwright self-click after run_app. Eliminates the App ▸ Settings ▸ Runtime ▸ After Startup soft-stop entirely — every clone build that needs seed data now has a self-contained autonomous path. Cross-references from §10 (layout-first) and §11 (theme module) so the agent finds it during clone-build planning.
Rules §2 recovery ladder updates New step 1.5: agent-level maia__health liveness check — saves a wait window when the WebSocket is dead. New step 3.5: between maia__reset and the user-handoff escalation, wipe Maia's panel state via maia__busy → wait-for-idle → maia__new_chat. Optional task-boundary new-chat guidance for keeping Maia sharp in long sessions.
New Codex + Copilot rules paths Each ticked CLI now ships Concord rules + a managed-block import file with identical content. claude → .claude/rules/ + CLAUDE.md; codex → .codex/rules/ + AGENTS.md; copilot → .github/rules/ + .github/copilot-instructions.md. Same fenced-block markers, same orphan-cleanup, same atomic write. The deferred Phase 2 from v4.1.4 lights up now that the Claude path is production-proven.
New Codex defaults-on + first-run banner Codex was opt-in through v4.2.0 because it writes to user-global ~/.codex/config.toml. v4.2.1 flips it to default-on with a first-run banner explaining the user-global write. Banner content: "Codex MCP wires ~/.codex/config.toml (user-global, outside the project tree). Untick Codex in Settings if you'd rather not." Informed default-on beats one-extra-click in the common case.
Fixed get_active_run_configuration after Settings save The action server rebuilt by saving Settings was missing two callbacks (run-config + project-info). After any Settings save, get_active_run_configuration returned "Active-run-configuration callback not wired". The auto-start path was already correct; v4.2.1 wires the same callbacks through the save path so the tool keeps working after a re-save.
Reliability Singleton CdpClient explicit dispose on toggle v4.2.0 left the previous CdpClient to GC when a Settings save replaced it. v4.2.1 disposes it explicitly (3-second bounded await on DisposeAsync) before swapping in the new one. Fixes the slow leak for users who toggle Maia on/off repeatedly.
Reliability JS-side scan throttle fix The 10-second CDP heartbeat now also triggers __maiaBridge.scan(). Defeats Chromium's WebView2 background-tab throttling that drifts Maia's internal scan timer toward seconds when Maia is hidden behind another pane. Persistent CDP socket is unaffected by tab visibility, so detection latency stays bounded at the heartbeat interval regardless of pane state.
New WebSocket test seam + reconnect tests v4.2.0's reconnect path was covered by manual smoke testing only. v4.2.1 adds an IWebSocketAdapter interface and a discovery-override seam so unit tests can simulate drop-on-Nth-call and verify reconnect-with-fresh-adapter without standing up a real Studio Pro CDP endpoint. Four new tests cover the failure paths that v4.2.0's production run hit.

Cross-platform note

Concord ships on Windows and macOS. Most features are platform-identical; two differences worth calling out:

  • Maia integration (and the four new introspection tools that build on it) is Windows-only — driving Maia from outside Studio Pro requires inspecting its WebView, which works on Windows (WebView2 + CDP) but isn't available on macOS (WKWebView requires host opt-in by Mendix). Concord ships a Mac-specific variant of the page-gen skill that hands the work to Maia through you instead of programmatically.
  • Hotkey-driven Studio Pro tools on Mac need Accessibility permission the first time — macOS prompts for it; you grant Studio Pro one-time access in System Settings → Privacy & Security → Accessibility.

Compatibility

  • Studio Pro: 11.10.0 or newer
  • Operating system: Windows 10 build 17763 (1809)+ · macOS 10.15 (Catalina)+
  • .NET runtime: 8.0 (bundled with Studio Pro 11)

Privacy

Concord is loopback-only — the Concord MCP server binds to 127.0.0.1; Maia integration uses Studio Pro's local WebView2 debug port. Nothing leaves your machine through Concord itself. The bundled skill packs and rules are static Markdown files copied into your project. The CLI agents you run inside Concord follow their own privacy contracts. The Diagnostic logging toggle (introduced in v4.2.0) and the four new introspection tools in v4.2.1 only operate against your local Studio Pro session — nothing is transmitted.
